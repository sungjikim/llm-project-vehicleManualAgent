🔍 실제 검색 전략 선택 로직

1단계: LLM이 Few-shot 예시를 통해 쿼리 분석 (general/specific/troubleshooting + hybrid_semantic/hybrid_balanced/hybrid_keyword)

2단계: 키워드 규칙이 LLM 결과를 오버라이드 (우선순위: 키워드 규칙 > LLM 분석)

3단계: 선택된 방법으로 검색 실행

📊 검색 방법별 특징

  • expanded_query: 전문 용어 확장 + 하이브리드 검색 (교체/문제/고장/이상 키워드)
  • multi_query: 3개 관점 쿼리 생성 + 벡터 검색 (복잡한 질문, 20자 이상 + '?')
  • hybrid_semantic: 의미론적 우선 (7:3) - LLM 분석 결과
  • hybrid_balanced: 균형 검색 (5:5) - LLM 분석 결과
  • hybrid_keyword: 키워드 우선 (3:7) - LLM 분석 결과

📋 실제 검색기 선택 예시 (수정된 정확한 표)

질문 유형 LLM 분석 결과 키워드 규칙 최종 선택 이유
"오일 교체 주기는?" general → hybrid_semantic 교체 키워드 감지 expanded_query 키워드 규칙이 우선
"브레이크 경고등 깜빡임" troubleshooting → hybrid_keyword 없음 hybrid_keyword LLM 결과 사용
"XC60 타이어 공기압" specific → hybrid_balanced 없음 hybrid_balanced LLM 결과 사용
"엔진 소음 이상" general → hybrid_semantic 이상 키워드 감지 expanded_query 키워드 규칙이 우선
"복잡한 긴 질문?" general → hybrid_semantic 복잡한 질문 감지 multi_query 키워드 규칙이 우선

🔧 주요 특징 (현재 구현 버전)

  • 모듈화된 아키텍처: 각 기능을 독립적인 모듈로 분리하여 재사용성 향상
  • LLM 기반 지능형 감지: 키워드 매칭이 아닌 맥락 이해로 정확한 응급/주행 상황 감지
  • 조건부 워크플로우: 응급 상황 감지 후 적절한 경로 자동 선택
  • 응급 상황 빠른 경로: CRITICAL/HIGH 응급 상황 전용 최적화된 처리
  • 하이브리드 검색: 벡터 검색과 BM25 키워드 검색의 지능적 조합
  • 쿼리 확장 및 다중 쿼리: 전문 용어 확장과 3개 관점 쿼리 생성
  • 주행 중 최적화: LLM 기반 주행 상황 감지 및 답변 압축
  • 실시간 모니터링: 성능, 토큰 사용량, 신뢰도 실시간 추적
  • 다중 인터페이스: 터미널과 Gradio 웹 인터페이스 지원
  • 음성 인식 지원: ASR/STT 기반 음성 입력 처리 (더미 모드 포함)
  • 통합 테스트 시스템: 다양한 시나리오를 통합한 테스트 프레임워크
